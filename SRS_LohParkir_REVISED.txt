1. INTRODUCTION
================================================================

1.1 Purpose
-----------
The purpose of LohParkir is to provide an official QR code-based parking validation system for the city of Medan. This system aims to solve three main problems: weak supervision of parking spots, lack of official identity that the public can easily verify, and no integrated system to document official parking data. By using this system, illegal parking attendants cannot easily demand money because every parking spot and officer has a verified QR code. If illegal parking still occurs, the public can immediately report it through the app with photo evidence and GPS location. The report is then sent to Dishub Medan for follow-up.

This SRS document describes all functional and non-functional requirements for the LohParkir system, version 1.1.


1.2 Document Conventions
-------------------------
This document follows these simple conventions:

- Bold text is used for menu names, button labels, or user interface elements. Examples: Scan QR, Laporkan, Dashboard Monitoring.

- Abbreviations are written in full when first used, followed by the abbreviation in parentheses. Example: Dinas Perhubungan (Dishub), Quality Assurance (QA).

- The word "user" refers to the general public who uses the mobile application to scan QR codes.

- The word "admin" refers to Dishub or Pemda (local government) staff who manage the system through the admin dashboard.

- The word "superadmin" refers to the Database Administrator who has full CRUD access to the system database.

1.4 Product Scope
------------------
LohParkir is designed specifically for official on-street parking managed by Dishub Medan. It does not cover parking inside buildings, malls, or private areas. The scope is limited to public roads where illegal parking often occurs.

The system includes the following main features:

Instant QR Code Validation: Users scan a QR code displayed at the parking location or on the officer's badge. The system checks the code against Dishub's official database. It then shows whether the parking is official or not, along with the location name, officer name, official rate, and work area.

Report Illegal Parking or Fake QR: If a scanned QR code is not registered in the database, the system shows a warning and provides a button to report it immediately. Users can submit a report with a photo, automatic GPS location, and a short description. The system then gives a ticket number and sends notifications about the follow-up status.

Digital Payment via QRIS: After validation, the app allows users to pay the official parking fee digitally through QRIS (Quick Response Code Indonesian Standard) or in cash. For digital payment, the system generates a QRIS code that users can scan using their mobile banking or e-wallet app to complete the transaction. All payments, including cash, are recorded in the system's database so that the government can track how much money was collected and where.

[REVISED - Issue #10: Clarified payment methods to QRIS + Cash only, consistent throughout]

Real-time Monitoring Dashboard: There is a dashboard that shows a summary of today's scans, number of active versus fake QR codes, trend graphs of validations, a heatmap of scan locations, and notifications of incoming reports. This dashboard is accessible to Dishub Medan staff with different roles (some can edit the database, others can only view). A public view is also available so that citizens can see parking validation statistics for transparency.

The system does not currently integrate with any external systems such as existing retribution systems or public complaint platforms. However, it is designed to be scalable for future integration.

The target users are all vehicle drivers in Medan who park on public streets. The number of expected users is very large, so the system must be able to handle high traffic.

Official QR codes for badges are generated by the government based on the registered database. The QR code format is: LOHPARKIR-DSH-YYYY-NNN (e.g., LOHPARKIR-DSH-2024-001). Officers must print and wear these QR codes as part of their official badge so that they can be scanned at any time.

2. OVERALL DESCRIPTION

2.1 Product Perspective
------------------------
LohParkir is a newly developed software system designed to support the management and validation of official parking services. The system is not a replacement for an existing automated system, but rather a digital enhancement of current manual or loosely monitored parking practices.

The system operates as part of a broader parking management ecosystem managed by the Department of Transportation (Dishub) or local government authorities (Pemda). Within this ecosystem, LohParkir functions as a validation and information layer that enables users to verify the authenticity of parking services through QR code scanning.

LohParkir is not a fully standalone system, as it depends on external data sources provided by Dishub/Pemda. Specifically, the system interfaces with an external database containing registered official QR codes. This interaction may involve data retrieval mechanisms such as APIs or other data exchange methods defined by the authority.

At a high level, the system consists of:
- A user application for scanning QR codes and displaying validation results
- A backend system for processing validation requests and managing data
- An administrative interface for Dishub/Pemda to manage QR code data and monitor system activity

The primary external interface of the system is the Dishub/Pemda data source, which provides the official QR code dataset used for validation. Additional interfaces may include network communication services required for real-time data exchange.

The system is limited to providing data validation and informational support. It does not include direct physical enforcement or monitoring of parking activities in the field.


2.2 Product Functions
----------------------
The LohParkir system provides a set of functions for four types of users: general public (drivers), Dishub admins, database administrators, and registered parking officers (who carry a QR badge and have access to a minimal officer app). The system also performs automatic functions. The main functions are grouped below.

2.5 Design and Implementation Constraints
-------------------------------------------
The following constraints apply to the development of LohParkir:

- The mobile app must be developed using React Native with Expo framework to support cross-platform deployment (Android and iOS) from a single codebase.
- The QR code format must follow the standard: LOHPARKIR-DSH-YYYY-NNN. Only Dishub-authorized admins or superadmins can generate new QR codes.
- All payment integrations must comply with Bank Indonesia's QRIS standard and PCI DSS requirements.
- The system must comply with Indonesia's Law No. 27 of 2022 on Personal Data Protection (UU PDP).
- The admin dashboard must be web-based to support access from standard office computers without requiring app installation.
- All API communications must use HTTPS with TLS 1.2 or higher.
- The system must use TypeScript for type safety across both frontend and backend codebases.
- Rate limiting must be implemented on all API endpoints to prevent abuse.

4.1.1 Description and Priority

This feature allows users to scan QR codes located at parking areas or on parking officer badges. The system validates the QR code against the Dishub/Pemda database and displays its authenticity status along with detailed information.

7. If invalid: The system displays a warning notification and provides a "Report" button to submit an illegal parking report.
8. If network error: The system displays a user-friendly error message and suggests retrying.

5. OTHER NONFUNCTIONAL REQUIREMENTS
================================================================

5.1 Performance Requirements
------------------------------
The system must respond quickly and handle many users at the same time. The following performance targets are required:

Response time (NFR-01): For normal actions like scanning a QR code, submitting a report, or loading a page, the system must respond within 2 seconds. If the user uploads a photo, it must respond within 3 seconds.

QR scan speed (NFR-02): The camera must detect and decode a QR code on an officer's badge within 1.5 seconds after the code enters the camera frame.

[REVISED - Issue #4: Changed from 1 second to 1.5 seconds for realism]

Dashboard load time (NFR-03): The admin dashboard must load real-time statistics within 3 seconds for initial load and within 1 second for data refresh.

[REVISED - Issue #5: Changed initial load from 1 second to 3 seconds for realism]

Concurrent users (NFR-04): The system must handle at least 500 users simultaneously and support up to 1,000 scans or reports per hour.

[REVISED - Issue #16: Changed from NFR-03 to NFR-04, unique ID]

Data growth (NFR-27): The system must be able to handle growth of up to 10,000 registered officers, 100,000 reports per year, and 500,000 scans per year without major changes.

[REVISED - Issue #8: Clarified metrics (officers, reports, scans instead of vague "users")]

Horizontal scaling (NFR-28): The backend must support scaling by adding more servers (horizontal scaling). It should use stateless APIs, be ready for a load balancer, and use containerized deployment.


5.2 Safety Requirements
-------------------------
The system must protect the physical and psychological safety of all users, including reporters, parking officers, and the general public:

Protection of reporter identity (NFR-10): The system must protect the identity of anyone who reports illegal parking or fake QR codes. Reporting must be anonymous. Reporter data must be encrypted, and only a superadmin can access it.

Prevention of report misuse (NFR-11): The system must prevent fake reports or spam that could harm honest officers. Each user can submit a maximum of 5 reports per day. Every report must include a photo (mandatory). All reports are moderated by an admin.

Payment transaction safety (NFR-12): Digital payments must be safe from double charging, failure without refund, or amount manipulation. The system must use an idempotency key for each transaction. If a payment fails, the system must automatically issue a refund. Every transaction must have a complete audit trail.

Location data protection (NFR-13): The system may collect the user's GPS location only during a scan or when submitting a report. It must not track the user continuously. The app must ask for permission before accessing location, and it should use one-time location (not background tracking). Location data must never be sold or shared with third parties.

Failsafe on critical errors (NFR-14): If a critical error occurs, the system must not show sensitive data, send duplicate payments, or delete data without confirmation. The system must degrade gracefully, avoid exposing technical details in error messages, and automatically roll back any failed transaction.

Physical safety of parking officers (NFR-15): The system must provide an emergency feature (panic button) for parking officers who face physical threats while on duty. The officer's minimal app must include a panic button. When pressed, the system sends a real-time notification to the admin and automatically shares the officer's GPS location.

[REVISED - Issue #11: Changed "officer's app" to "officer's minimal app" for consistency with Section 2.2]


5.3 Security Requirements
---------------------------
The system must protect data and prevent fraud. The following security requirements apply:

Admin authentication (NFR-05): Admin logins must use encrypted passwords (bcrypt hashing) and JWT tokens with expiry. The session will time out after 30 minutes of inactivity.

Data encryption (NFR-06): All communication between the mobile app and the server must use HTTPS with TLS 1.2 or higher. Sensitive data stored on the server must be encrypted using AES-256.

QR code integrity (NFR-07): Each QR code on an officer's badge must follow the fixed format: LOHPARKIR-DSH-YYYY-NNN (e.g., LOHPARKIR-DSH-2024-001). The system must validate each code using checksum verification to prevent forgery and duplication.

[REVISED - Issue #9: Changed from "LOH/PARKIR-DSH-YYY-NNN" to "LOHPARKIR-DSH-YYYY-NNN"]

Input validation (NFR-08): Every input from users (text, photos, reports) must be validated on the server side to prevent SQL injection and XSS attacks. The system must implement rate limiting of 100 requests per minute per user for general API endpoints.

[REVISED - Issue #7: Changed from 10 req/min to 100 req/min]

Authorization & RBAC (NFR-09): The system must use role-based access control (RBAC). There are three roles: public (regular users), admin (Dishub staff), and superadmin (database manager). Each API endpoint must check the user's role before allowing access.

Personal data protection (NFR-32): The system must follow Indonesia's Law No. 27 of 2022 on Personal Data Protection (UU PDP). This means collecting only necessary data, getting user consent, and handling data securely.

Data retention (NFR-33): Reports must be stored for at least 5 years. Payment transactions for at least 10 years. Scan logs for at least 1 year.

Digital payment regulation (NFR-34): The QRIS payment integration must follow Bank Indonesia standards and be PCI DSS compliant. The system must perform daily reconciliation of payments.


5.4 Software Quality Attributes
---------------------------------
These attributes describe how well the system works from the user's and developer's point of view.

Usability:

Ease of use (NFR-20): The app must be simple. A user should need no more than 3 taps to scan a QR code, 5 taps to send a report, and 4 taps to make a payment.

Language & localization (NFR-21): The entire user interface must be in Indonesian using clear and simple words. Font size must be at least 14sp.

Accessibility (NFR-22): The app must meet WCAG 2.1 Level AA standards. This includes a contrast ratio of at least 4.5:1 and touch targets of at least 44x44 dp.

Visual feedback (NFR-23): Every user action must show a loading indicator, a success or error message (toast/alert), and color-coded status.

Compatibility:

Platform support (NFR-24): The mobile app must run on Android 8.0 (API 26) or higher and iOS 14.0 or higher. It should look and work the same on both platforms. The framework used is React Native with Expo.

[REVISED - Issue #2: Changed "iOS 14.9" to "iOS 14.0"]

Screen size (NFR-25): The app must work on smartphone screens from 4.7 inches to 6.9 inches in portrait mode.

Camera compatibility (NFR-26): The QR scanner must work with cameras as low as 5 MP, support autofocus, and tolerate low-light conditions.

Maintainability:

Code quality (NFR-29): The source code must use TypeScript with strict mode, follow ESLint and Prettier rules, and require code reviews before merging.

Logging & monitoring (NFR-30): The system must use structured logging (JSON format) for debugging. All admin actions must be saved in an audit trail. Error tracking tools (e.g., Sentry) should be used.

API versioning (NFR-31): The backend API must use URL versioning (e.g., /api/v1/...). If a version is deprecated, users must be notified at least 3 months in advance.

Reliability & Availability:

Uptime (NFR-16): The system must be available 99.5% of the time during operational hours (6:00 AM to 10:00 PM WIB) and 24/7 for reporting. Maximum allowable downtime is approximately 3.6 hours per month.

[REVISED - Issue #6: Changed from 99.9% to 99.5% to match 3.6 hours/month downtime]

Offline mode (NFR-17): The mobile app must cache data when there is no internet connection. Users can save reports locally, and the app will automatically sync them when back online using a queue.

Backup & recovery (NFR-18): The system must perform automatic daily backups. Recovery point objective (RPO): 24 hours. Recovery time objective (RTO): 4 hours.

Error handling (NFR-19): The system must show user-friendly error messages without technical details. All errors must be logged on the server, and the app should degrade gracefully.


5.5 Business Rules
--------------------
Business rules are the logical constraints that guide how the system operates:

Reporting limits (BR-01, ref: NFR-11): A regular user cannot submit more than 5 reports per day to prevent spam. Each report must include a photo. All reports are moderated by an admin.

[REVISED - Issue #18: Changed from NFR reference to BR (Business Rule) reference]

Report follow-up (BR-02): Once a report is submitted, it cannot be deleted by the user. An admin must change its status (e.g., "received", "investigating", "resolved", "rejected"). The user will receive a notification when the status changes.

[REVISED - Issue #18: Removed incorrect "NFR-33" reference, assigned BR-02]

QR code issuance (BR-03, ref: NFR-07): Only a superadmin or a designated Dishub admin can generate official QR codes for new parking officers. Each QR code must be unique and follow the format LOHPARKIR-DSH-YYYY-NNN.

[REVISED - Issue #9: Consistent QR format; Issue #18: Added BR reference]

Payment recording (BR-04, ref: NFR-12, NFR-34): All payments (digital via QRIS and cash) must be recorded in the database. For cash payments, the officer or admin must enter the amount into the system at the end of each shift. The system must produce a daily reconciliation report. Every transaction must have an idempotency key and audit trail.

Data retention rules (BR-05, ref: NFR-33):
- Illegal parking reports: stored for 5 years.
- Payment transactions: stored for 10 years (for audit purposes).
- Scan logs (who scanned which QR code): stored for 1 year, then archived.

[REVISED - Issue #18: Proper BR reference]

Public dashboard access (BR-06): Any citizen can view the public dashboard (statistics, heatmap, total scans) without logging in. However, detailed reports and personal data are hidden.

Officer account registration (BR-07): A Dishub admin registers a new parking officer in the system and generates an official QR badge. The officer receives the badge and the minimal officer app. This prevents fake officers from self-registering.

[REVISED - Issue #12: Changed from self-registration to admin-only registration for clarity and consistency with Section 2.2]

Report immutability (BR-08): Users cannot edit a report after submission. If they made a mistake, they must submit a new report and mention the previous ticket number.

[REVISED - Issue #18: Removed incorrect "NFR-19" reference, assigned BR-08]

Session timeout (BR-09, ref: NFR-05): Admin users will be automatically logged out after 30 minutes of inactivity. Regular users (drivers) do not have a session timeout because they use the app without logging in.

Emergency button for officers (BR-10, ref: NFR-15): Only registered parking officers who have the official minimal app can use the panic button. The system must prioritize this alert and notify admins immediately.

[REVISED - Issue #11: "official app" → "official minimal app"]

Rate limiting for API (BR-11, ref: NFR-08): Each user (by IP or account) is limited to 100 requests per minute to prevent abuse.

[REVISED - Issue #7: Changed from 10 req/min to 100 req/min]

6. OTHER REQUIREMENTS
================================================================

Internationalization: The initial release will support Indonesian (Bahasa Indonesia) only. Future versions may add English language support.

Legal Compliance: The system must comply with all applicable Indonesian regulations, including but not limited to:
- UU No. 27 Tahun 2022 (Personal Data Protection)
- Bank Indonesia QRIS standards
- Local parking regulations (Perda Kota Medan)
