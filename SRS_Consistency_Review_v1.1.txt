--------------------------------------------------------------
ISSUE #1 — Officer Cash Recording Contradicts Officer Access
--------------------------------------------------------------
Severity: CRITICAL
Sections: 2.2 A, 2.3, 4.3.1, REQ-20

PROBLEM:
- Section 2.2 A (General Public functions) states: "pay in cash
  (recorded by the officer)."
- Section 4.3.1 states: "Cash payments are also supported and
  recorded by the officer or admin into the system."
- REQ-20 states: "The system shall support cash payment recording
  by officers or admins."

BUT Section 2.3 (Registered Parking Officers) explicitly states:
  "Access rights: Emergency panic button only. No access to
  system data, reports, or dashboards."

This is a direct contradiction. Officers are supposed to record
cash payments, yet their defined access rights limit them to only
the panic button.

RECOMMENDATION:
Either (A) expand officer access rights in Section 2.3 to include
a "Record Cash Payment" function in the minimal officer app, and
add a corresponding system feature or sub-feature; or (B) change
all references to state that cash is recorded by Dishub admins
at the end of the shift (not by officers directly).


--------------------------------------------------------------
ISSUE #2 — Push Notifications Priority vs Safety-Critical Content
--------------------------------------------------------------
Severity: CRITICAL
Sections: 4.8.1, NFR-15, BR-10

PROBLEM:
- Feature 4.8 (Push Notifications) is listed as Priority: Medium.
- However, this feature contains the panic button notification
  system (REQ-47, REQ-48), which is classified as safety-critical
  in NFR-15 and BR-10.
- NFR-15 states: "When pressed, the system sends a real-time
  notification to the admin and automatically shares the officer's
  GPS location."
- BR-10 states: "Alert prioritized, admins notified immediately."

A Medium-priority feature cannot be the delivery mechanism for a
safety-critical function. If Push Notifications is deprioritized
or deferred, the officer panic button becomes non-functional.

RECOMMENDATION:
Either (A) raise Push Notifications to Priority: High; or
(B) separate the panic button notification into its own
High-priority feature (e.g., 4.9 Emergency Alert System) and
keep general push notifications as Medium.


--------------------------------------------------------------
ISSUE #3 — RBAC Defines 3 Roles but System Has 4 User Classes
--------------------------------------------------------------
Severity: CRITICAL
Sections: 2.3, NFR-09

PROBLEM:
- Section 2.3 defines 4 user classes: General Public, Registered
  Parking Officers, Dishub Admin, Database Administrator.
- NFR-09 (Authorization & RBAC) states: "Three roles: public,
  admin, superadmin."
- Officers are not included as a role in RBAC, yet they have their
  own app with specific access (panic button, and potentially cash
  recording per Issue #1).
- If officers are not a recognized role, the system cannot enforce
  access control on the officer app — anyone could install it and
  trigger false panic alerts.

RECOMMENDATION:
Update NFR-09 to define 4 roles: public, officer, admin,
superadmin. Describe the officer role's permissions (panic button,
and optionally cash recording). This ensures RBAC actually
protects the officer app functions.

--------------------------------------------------------------
ISSUE #4 — Superadmin Functions Have No System Feature in Sec. 4
--------------------------------------------------------------
Severity: MAJOR
Sections: 2.2 C, Section 4

PROBLEM:
Section 2.2 C lists three Superadmin (Database Administrator)
functions:
  1. Full Data Management (CRUD on QR codes, locations, officers,
     tariffs)
  2. User Role Management (assign/modify admin roles)
  3. System Audit (review audit trails and system logs)

None of these have a corresponding System Feature in Section 4.
There are no REQs covering superadmin CRUD, role assignment, or
audit log viewing. These functions are described but never
specified as requirements.

RECOMMENDATION:
Add a new system feature, e.g., "4.9 System Administration
(Superadmin)" with sub-sections for data management, role
management, and audit trail viewing. Assign REQ IDs (REQ-49
through REQ-5x) to each functional requirement.

ISSUE #5 — "Print Reports" Function Has No REQ
--------------------------------------------------------------
Severity: MAJOR
Sections: 2.2 B, Section 4.4, Section 4.7

PROBLEM:
Section 2.2 B (Dishub Admin functions) states:
  "View and Print Reports: Generate and print monthly or custom
  reports on validation, payments, and illegal reports."

However, no REQ in Section 4 covers the print/export capability.
Feature 4.4 (Dashboard) only covers viewing statistics, and
Feature 4.7 (Report Management) only covers managing report
statuses. Neither includes requirements for generating printable
reports or exporting data.

RECOMMENDATION:
Add REQs for report generation and printing. This could be added
to Feature 4.4 or 4.7, e.g.:
  REQ-49: The system shall allow admins to generate summary
  reports for a selected time period.
  REQ-50: The system shall allow admins to export or print
  reports in PDF format.

ISSUE #7 — Offline Mode Not Covered by Any REQ
--------------------------------------------------------------
Severity: MAJOR
Sections: 2.4, NFR-17, Section 4

PROBLEM:
- Section 2.4 (Operating Environment) states: "Offline mode
  available for cached data and queued reports."
- NFR-17 states: "App caches data offline. Reports saved locally
  and auto-synced when back online."
- However, no functional requirement (REQ) in Section 4 specifies
  offline behavior. Feature 4.2 stimulus/response (reporting)
  assumes online connectivity throughout. REQ-15 says "The system
  shall submit reports to the server" with no offline fallback.

Without a REQ, developers have no specification for how offline
mode should work (what is cached, how sync works, conflict
resolution, etc.).

RECOMMENDATION:
Add offline-related REQs, either within Feature 4.2 or as a
cross-cutting concern:
  REQ-xx: The system shall allow users to save reports locally
  when no internet connection is available.
  REQ-xx: The system shall automatically submit locally saved
  reports when internet connectivity is restored.
  REQ-xx: The system shall display cached validation data when
  offline.

ISSUE #8 — Product Scope Lists 4 Features but Section 4 Has 8
--------------------------------------------------------------
Severity: MAJOR
Sections: 1.4, Section 4

PROBLEM:
Section 1.4 (Product Scope) lists 4 "main features":
  1. Instant QR Code Validation
  2. Report Illegal Parking or Fake QR
  3. Digital Payment via QRIS
  4. Real-time Monitoring Dashboard

But Section 4 (System Features) defines 8 features:
  4.1 QR Code Validation Scan
  4.2 Report Illegal Parking
  4.3 Integrated Digital Payment (QRIS)
  4.4 Real-time Monitoring Dashboard
  4.5 Officer Account & QR Badge Management
  4.6 Payment History
  4.7 Report Management
  4.8 Push Notifications

Features 4.5, 4.6, 4.7, and 4.8 are not mentioned in the scope.
While 1.4 briefly alludes to some (e.g., "QR codes for badges are
generated by the government"), readers expect the scope to provide
a complete overview of all system features.

RECOMMENDATION:
Either expand Section 1.4 to briefly mention all 8 features, or
add a summary statement like: "In addition to the four core
features above, the system also includes Officer Account & QR
Badge Management, Payment History, Report Management, and Push
Notifications (detailed in Section 4)."

--------------------------------------------------------------
ISSUE #11 — Report Stimulus Does Not Cover Standalone Reporting
--------------------------------------------------------------
Severity: MINOR
Sections: 4.2.2

PROBLEM:
Feature 4.2 stimulus/response sequence starts with:
  "1. The system detects an invalid or inactive QR code."

This implies that reports can only be submitted AFTER a failed QR
scan. However, Section 1.4 and 2.2 A describe the report feature
more broadly: users should be able to report "illegal parking"
(e.g., unauthorized parking attendants operating without any QR
code at all). In real-world usage, a user may want to report
illegal parking they observe without scanning anything first.

RECOMMENDATION:
Add an alternative stimulus path, e.g.:
  "1a. The user navigates to the Report section directly from the
  home screen (without scanning)."
This ensures the report function is not locked behind a failed
QR scan.
